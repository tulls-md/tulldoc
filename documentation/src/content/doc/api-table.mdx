# Таблица API

Секция **API** генерируется автоматически для каждого `.doc.tsx`-документа.
Tulldoc анализирует TypeScript-тип первого параметра компонента через compiler API
и строит таблицу пропсов без какого-либо ручного описания.

## Что показывается

| Колонка      | Источник                                      |
| ------------ | --------------------------------------------- |
| Имя          | Имя свойства TypeScript-типа                  |
| Тип          | Текст типа (нормализованный)                  |
| Обязательный | Свойство без `?`                              |
| По умолчанию | Деструктуризация параметра или тег `@default` |
| Описание     | JSDoc-комментарий над свойством (markdown)    |

## JSDoc-описания

Добавляйте комментарии прямо над свойствами интерфейса:

```ts
interface ButtonProps {
  /** Визуальный стиль. Влияет на **цвет** и обводку, см. `size`. */
  variant?: "primary" | "secondary" | "ghost";

  /**
   * Блокирует взаимодействие и применяет стиль недоступности.
   * Форма продолжает принимать значение поля.
   */
  disabled?: boolean;

  children: React.ReactNode;
}
```

В описаниях поддерживается markdown: `` `код` ``, **жирный**, _курсив_, ссылки
и списки. Заголовки и блоки кода деградируют до абзацев - вёрстка таблицы
не ломается. Упоминание другого пропа в инлайн-коде (`` `size` ``)
автоматически становится ссылкой на его строку таблицы.

## Значения по умолчанию

Tulldoc читает значения по умолчанию из деструктуризации параметра функции:

```ts
export function Button({
  variant = "primary",
  size = "md",
  disabled = false,
  children,
}: ButtonProps) {
  /* ... */
}
```

В таблице появятся: `variant` -> `"primary"`, `size` -> `"md"`, `disabled` -> `false`.
Если пропа нет в деструктуризации, используется JSDoc-тег `@default`
(или `@defaultValue`); деструктуризация важнее - это реальный рантайм-дефолт.

## Устаревшие пропсы

Проп с JSDoc-тегом `@deprecated` показывается зачёркнутым с бейджем
«устаревший»; текст тега добавляется к описанию:

```ts
interface ButtonProps {
  /**
   * Произвольный контент после текста.
   *
   * @deprecated Используйте `label`
   */
  extra?: React.ReactNode;
}
```

## Якоря строк

Каждая строка таблицы получает стабильный `id` вида
`props-<компонент>-<проп>`: например, `Card.Root` и проп `isLoading` ->
`#props-card-root-isloading`. На строку можно ссылаться из описаний;
упоминания пропов в инлайн-коде линкуются на эти якоря автоматически.

## Сборные компоненты: subcomponents

Для компонента из нескольких частей объявите `subcomponents` в `DocMeta` -
после основной таблицы появится отдельная таблица на каждый подкомпонент
(с заголовком в оглавлении):

```ts
return {
  component: Card.Root,
  componentName: "Card.Root",
  subcomponents: [
    {
      component: Card.Header,
      componentName: "Card.Header",
      description: "Шапка карточки: `title` и `subtitle`.",
    },
    { component: Card.Content, componentName: "Card.Content" },
  ],
};
```

Автопримеры генерируются только из пропов главного компонента -
подкомпоненты добавляют только таблицы API.

## Нормализация типов

Типы перед отображением проходят нормализацию:

- `React.ReactNode` -> `ReactNode` (убирается префикс `React.`)
- Однослойный не-экспортируемый алиас разворачивается: `type Size = "sm" | "md"` -> `"sm" | "md"`
- Типы из `node_modules` остаются по имени алиаса
- Лишние пробелы нормализуются

```ts
// ButtonProps.tsx
type InternalSize = "sm" | "md" | "lg"; // не экспортируется -> разворачивается

export interface ButtonProps {
  size?: InternalSize; // покажет: "sm" | "md" | "lg"
}
```

## Порядок пропсов

1. **Собственные пропсы** - объявленные в том же файле, что и компонент
2. **Унаследованные пропсы** - из других файлов (например, из `HTMLButtonElement`)

Это позволяет собственным пропсам всегда оказываться вверху таблицы.

## Пустая таблица

Если компонент не принимает пропсов (или параметр не найден), таблица заменяется текстом:

> Компонент не принимает пропсов.

## Требования к componentsDir

Таблица API работает только при наличии `componentsDir` в плагине `componentDocs`
(аддон `@tulls-md/tulldoc-code`). Без него tulldoc не может найти файл компонента для анализа.

```ts
createDocSource({
  // ...
  plugins: [
    componentDocs({
      componentsDir: join(process.cwd(), "../../packages/ui/src/components"),
      // ...
    }),
  ],
});
```

Tulldoc ищет файл компонента в `componentsDir`, опираясь на имя функции (или `componentName`).
